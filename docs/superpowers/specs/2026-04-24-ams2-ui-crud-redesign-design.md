# AMS2.0 UI/UX 重设计与基础资料 CRUD 改造设计

## 1. 背景

当前 AMS2.0 已经具备最小可运行能力：

- 登录
- 用户、部门、资产树、申请列表、申请新建
- 设备权限查询
- 审计日志
- Docker 部署

但当前版本仍存在两个核心问题：

1. 前端页面过于简化，整体更接近联调页或原型页，缺少成熟业务系统应有的视觉层级、操作节奏和统一交互框架。
2. 多数页面仍然以只读展示为主，缺少管理员高频使用所需的新增、编辑、删除闭环。

本次设计的目标不是“把页面美化一下”，而是将系统升级为真正可运营的“审批驱动账号与权限管理中台”。

## 2. 设计目标

本轮改造要同时达成以下目标：

- 将整体视觉升级为专业中后台产品，而非简单表格页面。
- 建立统一的工作台式操作框架，使高频业务操作可预测、可复用。
- 让基础资料对象具备完整 CRUD 能力。
- 保持权限敏感动作继续走审批流，避免管理员直接越权修改权限。
- 让申请、查询、审计页与新视觉体系统一，但不在第一轮引入过量复杂图表。
- 确保桌面端操作效率优先，同时保证移动端和小屏下的可读性。

## 3. 本轮范围

### 3.1 纳入本轮改造的页面

- 登录页
- 工作台
- 用户管理
- 部门管理
- 设备账号管理
- 申请列表
- 新建申请
- 查询页（统一视觉和交互骨架）
- 审计页（统一视觉和交互骨架）

### 3.2 本轮纳入 CRUD 的对象

- 用户
- 部门
- 设备账号

### 3.3 本轮明确不做管理员直改的动作

以下动作必须继续通过审批流完成，不允许在管理页直接提交最终变更：

- 给某人增加某设备角色
- 删除某设备角色
- 重置设备账号密码
- 角色变更类操作
- 批量授权类操作

这些动作可以从管理页发起，但最终应转化为预填申请单，并进入既有审批链。

## 4. 当前系统能力与改造边界

### 4.1 当前前端现状

前端目前采用 Vue 3 + Pinia + Vue Router + Element Plus，但页面层存在以下问题：

- `AppLayout` 仅有极简头部和内容区，没有真正导航系统。
- 工作台只是链接列表，不具备运营驾驶舱属性。
- 管理页以只读表格/树为主，没有统一的工具栏、筛选区、编辑工作流。
- 页面之间视觉风格过于平，缺少状态强调和操作层级。

### 4.2 当前后端现状

后端当前能力大致为：

- `users`: 列表查询
- `departments`: 列表查询
- `assets`: 资产树查询
- `device-accounts`: 按设备查询账号
- `requests`: 列表 + 新建
- `queries`: 设备权限查询
- `audit`: 审计列表

即：基础资料 CRUD 能力尚未铺开，现阶段前端无法真正完成用户提出的“增删改”要求。

### 4.3 本轮边界

本轮只对“基础资料维护闭环”和“系统级 UI/UX 架构升级”负责，不重构审批状态机本身。

换句话说：

- 审批流继续沿用现有主链路
- 申请单作为独立业务页继续存在
- 本轮主要补齐管理台能力和交互框架

## 5. 视觉方向

本轮采用已确认的 **B 方案：Operations Cockpit（业务驾驶舱）**。

### 5.1 视觉关键词

- 专业中台
- 业务驾驶舱
- 高信息密度但不拥挤
- 强工具栏和状态反馈
- 适度品牌感，不做花哨视觉噪音

### 5.2 视觉特征

- 左侧固定导航栏，深色或深灰蓝底，建立稳定的产品骨架。
- 顶部命令栏承载全局搜索、快捷操作、待办和登录用户信息。
- 主内容区使用浅暖白与冷灰蓝混合的分层背景，避免纯白表格系统的廉价感。
- 指标卡、状态标签、审批步骤条、操作按钮使用高可读且有业务语义的颜色系统。
- 页面首屏既要有系统感，也要有“能立刻操作”的效率感。

### 5.3 视觉非目标

- 不走营销官网风格
- 不做炫技式图表大屏
- 不使用夸张渐变和装饰性动画
- 不做过于极简的留白型管理台

## 6. 信息架构

### 6.1 顶层导航结构

建议左侧导航固定为：

- 工作台
- 用户管理
- 部门管理
- 设备账号管理
- 申请管理
- 查询管理
- 审计追踪

### 6.2 顶部命令栏

顶部命令栏统一包含：

- 全局搜索入口
- 快捷新建入口
- 待处理申请/待审批提示
- 当前登录用户
- 退出登录

### 6.3 主工作区模型

大多数页面采用统一主模型：

- 页面标题区
- 指标/摘要区
- 工具栏
- 筛选区
- 主数据表格
- 右侧抽屉详情/编辑区

申请相关页面例外，采用独立主页面承载完整流程信息。

## 7. 页面级设计

## 7.1 登录页

登录页不再只是两行输入框，而是承担以下职责：

- 体现系统品牌和可信度
- 告诉用户系统是做什么的
- 让首次进入用户感受到“这是一个正式业务系统”

页面结构建议：

- 左侧/上方为系统品牌说明
- 右侧/下方为登录表单
- 表单区包含登录名、密码、登录按钮、错误反馈
- 背景可使用稳重的层次渐变或轻度纹理，而不是纯白

## 7.2 工作台

工作台从“链接汇总页”升级为真正的运营驾驶舱。

建议内容：

- 顶部关键指标卡
  - 用户总数
  - 部门总数
  - 设备账号总数
  - 待处理申请数
- 中部近期申请列表
- 中部待审批/待处理任务
- 侧边或下方异常提示
  - 账号未绑定人员
  - 设备账号缺角色
  - 即将需要处理的申请

工作台不做复杂 BI，而是聚焦“进入系统后第一眼要处理什么”。

## 7.3 用户管理页

用户页采用“表格 + 抽屉编辑”模式。

主表格字段建议包含：

- 人员编码
- 员工姓名
- 登录名
- 部门名称
- 在职状态
- 账号状态

工具栏动作：

- 新增用户
- 批量停用
- 批量筛选

行内动作：

- 编辑
- 删除
- 查看关联设备账号

抽屉表单字段：

- 人员编码
- 员工姓名
- 登录名
- 部门
- 在职状态
- 账号状态

## 7.4 部门管理页

部门页也采用统一管理台模式。

主表格字段建议包含：

- 部门名称
- 部门负责人
- 成员数量
- 最近更新时间

工具栏动作：

- 新增部门
- 筛选

行内动作：

- 编辑
- 调整负责人
- 删除

抽屉表单字段：

- 部门名称
- 部门负责人
- 部门说明（可选）

## 7.5 设备账号管理页

设备账号页是本轮最重要的核心操作台。

建议其结构高于普通列表页，具备以下能力：

- 按设备筛选
- 按人员筛选
- 按账号名筛选
- 查看角色绑定
- 编辑账号基础信息
- 绑定/解绑人员
- 删除账号
- 发起权限相关申请

主表格字段建议包含：

- 设备名称
- 账号名
- 绑定人员
- 当前角色
- 最近更新时间

工具栏动作：

- 新增账号
- 筛选器

行内动作：

- 编辑资料
- 删除账号
- 发起加权申请
- 发起删权申请
- 发起密码重置

右侧抽屉负责基础资料维护：

- 账号名
- 所属设备
- 绑定人员
- 备注

权限动作不在抽屉直接执行，而是跳转/打开预填申请流程。

## 7.6 申请列表页

申请列表页保留独立主页面，但视觉结构与业务驾驶舱统一。

建议增加：

- 状态筛选
- 申请类型筛选
- 创建时间筛选
- 申请状态标签
- 快速查看申请详情

这类页面不适合走右侧抽屉主编辑模式，但可用抽屉查看摘要。

## 7.7 新建申请页

新建申请页保持独立主页面，重点强调：

- 步骤感
- 表单清晰度
- 审批链可见性
- 提交结果反馈

建议页面包含：

- 顶部申请步骤条
- 左侧或主体申请表单
- 右侧审批链说明
- 提交后成功反馈及跳转入口

## 7.8 查询页与审计页

第一轮不引入复杂图表，以统一视觉和筛选效率为主。

查询页重点：

- 统一筛选区
- 表格结果可读性
- 空状态和异常提示

审计页重点：

- 时间、人员、设备、动作筛选
- 结果列表密度提升
- 操作对象与动作语义清楚

## 8. CRUD 与审批联动规则

### 8.1 可直接 CRUD 的对象

#### 用户

- 新增
- 编辑
- 停用
- 删除

#### 部门

- 新增
- 编辑
- 调整负责人
- 删除

#### 设备账号

- 新增
- 编辑
- 删除
- 绑定/解绑人员

### 8.2 必须走审批流的动作

- 权限新增
- 权限删除
- 密码重置
- 角色变更
- 批量授权

### 8.3 联动方式

在设备账号页或相关页面点击敏感动作时：

- 不直接修改目标数据
- 生成预填申请页或预填申请数据
- 用户确认后生成申请单
- 跳转到申请列表或申请详情

## 9. 业务保护规则

为避免直接 CRUD 破坏业务数据，本轮必须在后端补充以下保护规则。

### 9.1 部门删除规则

- 若部门下仍有成员，则禁止删除
- 必须先迁出成员或重新分配部门
- 返回明确错误提示，不能返回通用 500

### 9.2 用户删除规则

- 若用户仍绑定设备账号或存在未完成申请，则不允许硬删
- 建议优先支持“停用”而非直接物理删除

### 9.3 设备账号删除规则

- 若账号仍有关联角色或仍被流程引用，则禁止删除
- 必须先解除业务占用

### 9.4 审批动作反馈规则

- 所有审批类动作提交后，页面需提示“已生成申请单”
- 提供跳转申请列表或申请详情的入口

## 10. 前端实现设计

## 10.1 壳层重构

前端需要建立统一的业务驾驶舱壳层，替代当前极简 `AppLayout`。

建议新增统一壳层能力：

- 左侧导航
- 顶部命令栏
- 页面容器统一边距与栅格
- 统一状态色、标签、按钮、卡片、表格、抽屉规范

## 10.2 页面骨架统一

建议将以下页面统一为相同骨架：

- 用户管理
- 部门管理
- 设备账号管理
- 查询页
- 审计页

统一骨架：

- 页面抬头
- 摘要指标
- 工具栏
- 筛选区
- 主表格
- 抽屉

## 10.3 表单抽屉模型

建议抽离 3 类抽屉表单：

- 用户表单抽屉
- 部门表单抽屉
- 设备账号表单抽屉

这样可避免每个页面自己拼接一套编辑交互，保证体验一致。

## 10.4 列表刷新策略

建议采用局部刷新模式：

- 初始进入加载列表
- 新增/编辑成功后关闭抽屉并刷新当前页数据
- 删除成功后局部刷新列表并维持当前筛选条件

## 10.5 错误反馈

前端统一区分三类错误：

- 表单校验错误
- 业务规则阻断
- 系统异常

错误提示应明确说明原因，例如：

- 部门下仍有成员，无法删除
- 用户仍绑定设备账号，建议先停用
- 当前设备账号仍有关联角色，无法删除

## 11. 后端实现设计

## 11.1 用户接口扩展

在现有 `users` 列表基础上补齐：

- 用户新增
- 用户编辑
- 用户停用/启用
- 用户删除

## 11.2 部门接口扩展

在现有 `departments` 列表基础上补齐：

- 部门新增
- 部门编辑
- 负责人调整
- 部门删除

## 11.3 设备账号接口扩展

在现有按设备查询接口基础上补齐：

- 新增设备账号
- 编辑设备账号
- 删除设备账号
- 绑定/解绑人员

## 11.4 返回字段统一

本轮要求后端统一字段语义，降低前端页面的映射复杂度。

例如：

- 用户统一返回 `userName / loginName / departmentName / accountStatus`
- 部门统一返回 `departmentName / managerName / memberCount`
- 设备账号统一返回 `deviceNodeId / deviceName / accountName / userName / roles`

避免相同概念在不同接口中出现多套命名。

## 11.5 错误语义

后端应为可预期业务错误返回明确 message，而非直接抛 500。

典型场景：

- 重复登录名
- 部门下仍有成员
- 用户存在关联设备账号
- 设备账号仍有关联角色
- 请求参数非法

## 12. 数据流设计

### 12.1 基础资料页

- 页面加载时读取列表
- 点击新增/编辑后打开抽屉
- 抽屉提交成功后刷新列表
- 删除成功后刷新列表并保留筛选条件

### 12.2 审批动作

- 当前页面点击“加权/删权/重置密码”
- 打开预填好的申请流程
- 用户确认提交
- 生成申请单
- 返回原页或跳转申请列表

### 12.3 查询页

- 用户输入筛选条件
- 查询结果以统一表格承载
- 必要时提供详情抽屉或辅助信息面板

## 13. 非功能要求

### 13.1 响应式

- 桌面端为主，保证高频操作效率
- 小屏时左侧导航可折叠
- 抽屉和表格在窄屏下可降级为全宽面板

### 13.2 一致性

- 按钮层级统一
- 状态标签统一
- 成功/失败反馈统一
- 列表和表单布局统一

### 13.3 可扩展性

壳层与页面骨架设计应便于后续继续扩展：

- 用户权限查询
- 个人权限历史
- 批量导入
- 更细粒度设备层级编辑

## 14. 测试策略

本轮继续采用 TDD。

### 14.1 后端测试

先补齐后端测试，再实现接口：

- 用户 CRUD 测试
- 部门 CRUD 与删除约束测试
- 设备账号 CRUD 与删除约束测试
- 审批敏感动作仍通过申请流的测试

### 14.2 前端测试

补充页面行为测试：

- 用户管理页抽屉新增/编辑/删除
- 部门管理页新增/编辑/删除
- 设备账号管理页新增/编辑/删除
- 审批动作按钮能正确跳转/生成预填申请
- 工作台与新壳层稳定渲染

### 14.3 集成验证

最终验证包括：

- `npm test`
- `npm run build`
- `mvn test`
- `docker compose up --build -d`
- 浏览器冒烟验证关键操作闭环

## 15. 第一轮验收标准

第一轮完成后，必须满足以下验收结果：

- 登录页与工作台完成 B 方案重设计
- 用户、部门、设备账号页面统一切换为工作台模式
- 用户可完成用户新增、编辑、停用/删除闭环
- 用户可完成部门新增、编辑、负责人调整、删除闭环
- 用户可完成设备账号新增、编辑、绑定/解绑人员、删除闭环
- 权限类操作不会直接修改数据，而是触发申请流程
- 查询页与审计页完成视觉统一
- Docker 部署环境下关键流程可验证

## 16. 非目标

本轮明确不做以下内容：

- 审批状态机重构
- 完整 BI 仪表盘
- 复杂图表分析中心
- 全量批量导入导出体系
- 权限模型全面重做
- 多租户或多组织架构扩展

## 17. 推荐实施顺序

建议实施顺序如下：

1. 建立统一壳层与视觉系统
2. 重做登录页与工作台
3. 补齐用户 CRUD 后端与前端
4. 补齐部门 CRUD 后端与前端
5. 补齐设备账号 CRUD 后端与前端
6. 将权限敏感动作接入申请页联动
7. 统一查询页与审计页视觉
8. 完成 Docker 冒烟与上线验证
