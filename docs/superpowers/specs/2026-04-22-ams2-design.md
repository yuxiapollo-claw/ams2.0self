# AMS2.0 系统设计文档

## 1. 文档信息

- 文档名称：AMS2.0 系统设计文档
- 文档版本：v1.0
- 编写日期：2026-04-22
- 设计范围：完整目标版
- 适用对象：产品、开发、测试、实施、运维

## 2. 设计目标与边界

AMS2.0 的目标是统一管理企业内部人员、部门、设备/系统、设备账号、角色授权、审批流转、实施反馈与审计追踪，形成可查询、可审批、可追溯的账号管理平台。

本期设计边界如下：

- AMS 登录认证采用独立登录模式，由系统自身维护用户名、密码与系统角色
- 设备账号管理采用流程管理型模式
- 审批通过后由 QI 线下到真实设备或系统中执行，不直接由 AMS 自动下发
- 审批流固定为：申请人所在部门负责人 -> QA -> QM -> QI
- QA、QM、QI 采用全局固定岗位模式
- 系统建设目标为完整目标版，而非最小 MVP

不在本期强制范围内的能力：

- 对接 AD、LDAP、SSO
- 自动调用设备接口进行账号开通、删权、重置密码
- 可配置流程引擎
- 微服务拆分

## 3. 总体架构设计

AMS2.0 采用 B/S 架构，推荐使用前后端分离方式建设，并以单体后端作为核心业务系统。

系统整体由以下六个核心业务域组成：

1. 组织与人员域
2. 设备与角色域
3. 设备账号域
4. 申请审批域
5. 实施与审计域
6. 查询分析域

架构定位如下：

- AMS2.0 是主数据、流程、留痕与查询的一体化平台
- AMS2.0 不是自动化运维平台
- 审批通过仅表示业务批准，QI 线下实施成功后才表示真实业务完成

## 4. 业务模块设计

### 4.1 用户管理

维护人员主数据，核心字段包括：

- 人员编码
- 员工姓名
- 所属部门
- 部门负责人
- 在职状态
- AMS 登录账号
- AMS 账号状态

人员是全系统唯一主身份，设备账号、审批记录、审计日志均基于人员主档关联。

### 4.2 部门管理

维护部门信息和负责人关系，至少包括：

- 部门名称
- 部门负责人
- 部门成员

部门负责人是审批流第一节点的关键数据，必须保存历史快照以支撑审批追溯。

### 4.3 系统管理

使用通用树形结构维护设备、软件、角色等层级关系。

典型示例：

- 仪器设备 -> 部门 -> 设备A -> 软件 -> 角色

设计原则：

- 不是每棵树都必须满五层
- 每条可授权路径的最后一级必须是角色层
- 设备树结构不能写死层级，应允许按父子节点灵活配置

### 4.4 账号管理

设备账号管理规则如下：

- 同一人员在同一设备上只能存在一个设备账号
- 该设备账号可与 AMS 登录账号不同
- 同一人员在不同设备上的设备账号可以不同
- 一个设备账号可绑定多个角色
- 管理员可按设备和角色批量为人员建立关联

核心关系不是“人直接拥有角色”，而是：

`人员 -> 设备账号 -> 角色集合`

### 4.5 审批流管理

所有影响设备账号和角色的操作必须通过固定审批流执行：

`部门负责人 -> QA -> QM -> QI`

系统负责：

- 申请单生成
- 状态流转
- 审批意见留存
- 当前待办识别
- 实施回填

### 4.6 账号操作

需要纳入统一申请流程的操作包括：

- 权限申请
- 权限删除
- 重置密码

### 4.7 审计追踪

系统需能够清晰记录：

- 某人
- 某时
- 对某设备或某申请单
- 做了什么动作
- 结果如何

### 4.8 查询管理

系统至少支持以下三类查询：

- 按设备查询：查询某设备下所有权限组及关联账号
- 按人员查询：查询某人员在所有设备上的角色及设备账号
- 按人员查询申请记录：查询某人员全部权限申请历史

## 5. 角色与权限模型

AMS2.0 建议定义以下系统角色：

- 系统管理员
- 普通申请人
- 部门负责人
- QA
- QM
- QI
- 审计/查询用户

权限分为两层：

### 5.1 AMS 系统功能权限

控制菜单、按钮、接口访问，例如：

- 是否可维护设备树
- 是否可审批
- 是否可提交实施结果
- 是否可导出审计日志

### 5.2 设备业务权限

控制某人员在某设备上的账号与角色关系。

这部分属于 AMS 管理的业务数据，不等同于 AMS 菜单权限。

## 6. 核心数据模型设计

数据库设计围绕以下四条主线展开：

- 人员与组织
- 设备与角色
- 账号与授权
- 申请与审计

### 6.1 主数据实体

- `sys_user`：人员主表
- `sys_department`：部门表
- `asset_node`：设备/系统树节点表

### 6.2 授权关系实体

- `device_account`：某人员在某设备上的唯一账号
- `device_account_role`：设备账号与角色绑定关系

### 6.3 申请审批实体

- `request_order`：申请单主表
- `request_order_item`：申请单明细表
- `request_approval_record`：审批记录
- `request_execution_record`：QI 实施记录

### 6.4 审计实体

- `audit_log`：审计日志主表
- `audit_log_detail`：审计明细表

### 6.5 关键模型规则

- 人员编码唯一
- AMS 登录名唯一
- 同一人员在同一设备上只能有一个设备账号
- 同一设备账号下同一角色不可重复绑定
- 设备、角色停用后不得被新申请引用
- 离职人员不得再发起新的权限申请
- 审批单应保存审批人快照

## 8. 关键业务流程设计

### 8.1 统一流程主线

所有账号操作共用统一主流程：

`申请发起 -> 部门负责人审批 -> QA 审批 -> QM 审批 -> QI 实施 -> 完成归档`

规则如下：

- 任一审批节点驳回，申请单结束
- 申请人在审批未完成前可撤回
- QM 通过后进入待实施
- QI 回填成功后才视为业务完成
- 审批通过不等于业务已生效

### 8.2 权限申请流程

适用于为某人员在某设备下新增一个或多个角色。

处理规则：

1. 选择目标人员、目标设备、目标角色
2. 检查目标人员在该设备上是否已有设备账号
3. 若无账号，则在申请中填写设备账号名
4. 若已有账号，则复用该账号
5. 提交后生成申请主单与角色明细
6. 审批通过后由 QI 线下实施
7. 实施成功后将角色关系置为生效

### 8.3 权限删除流程

适用于删除某设备下某人员的部分角色或整套设备账号权限。

规则：

- 删除角色时保留设备账号
- 删除设备账号需保证其下已无有效角色，或经业务明确允许整账号注销

### 8.4 重置密码流程

适用于重置某人员在某设备上的账号密码。

规则：

- 审批通过后由 QI 线下重置
- AMS 仅记录已重置及实施说明
- 不存储任何明文密码

### 8.5 批量加人流程

管理员按设备和角色批量勾选人员后，生成批量申请。

系统要求：

- 自动识别已有设备账号与需新建账号的人员
- 统一走审批流
- 支持部分成功、部分失败

## 9. 状态机与异常处理

### 9.1 主单状态

- `DRAFT`
- `WAIT_DEPT_MANAGER`
- `WAIT_QA`
- `WAIT_QM`
- `WAIT_QI_EXECUTE`
- `COMPLETED`
- `REJECTED`
- `WITHDRAWN`
- `EXECUTE_FAILED`
- `PARTIAL_COMPLETED`

### 9.2 明细状态

- `PENDING`
- `APPROVED`
- `WAIT_EXECUTION`
- `EXECUTION_SUCCESS`
- `EXECUTION_FAILED`
- `CANCELLED`

### 9.3 允许动作矩阵

- `DRAFT`：编辑、删除、提交
- `WAIT_DEPT_MANAGER`：部门负责人通过/驳回，申请人撤回
- `WAIT_QA`：QA 通过/驳回，申请人撤回
- `WAIT_QM`：QM 通过/驳回，申请人撤回
- `WAIT_QI_EXECUTE`：QI 提交实施结果
- `EXECUTE_FAILED`：QI 再次提交实施结果，管理员处理异常
- `PARTIAL_COMPLETED`：QI 补交剩余实施结果，管理员确认结单
- `COMPLETED/REJECTED/WITHDRAWN`：只读

### 9.4 异常处理规则

- 重复申请：默认禁止，提示已有在途单据
- 审批中组织或岗位变化：以申请发起时快照为准
- 审批中目标人员离职：进入异常待处理，默认不继续推进
- 审批中设备或角色停用：不允许继续实施
- QI 线下实施失败：不重走审批，停留实施阶段重试
- 批量申请部分失败：主单记为部分完成，明细逐条标记

## 13. 安全设计

### 13.1 认证与登录

- 用户名 + 密码登录
- 密码采用强散列保存
- 支持首次登录强制改密
- 支持错误次数锁定
- 支持账号启停
- 支持会话超时

### 13.2 权限控制

- 功能权限控制菜单、按钮、接口
- 数据权限控制可查看的数据范围

建议基础数据权限策略：

- 普通申请人仅查看自己发起的单据
- 部门负责人查看本部门相关申请
- QA/QM/QI 查看职责范围内待办和历史
- 系统管理员查看全量数据

### 13.3 审计要求

以下动作必须审计：

- 登录成功/失败
- 用户、部门、设备树维护
- 设备账号与角色变更
- 申请提交、撤回
- 每个审批节点操作
- 每次实施回填
- 导出操作

## 14. 部署设计

目标环境：

- 应用服务器：Ubuntu，`10.1.18.53`
- 数据库服务器：MySQL，`10.1.18.47`

推荐部署方式：

- 前端构建后由 Nginx 托管静态资源
- 后端 Spring Boot 服务部署在应用服务器
- 后端通过内网访问数据库
- Nginx 反向代理 `/api`
- 日志按天切分
- 数据库执行每日备份
