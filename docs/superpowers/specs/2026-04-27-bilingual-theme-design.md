# AMS2.0 双语与主题切换设计

## 背景

当前 AMS2.0 前端已经完成 Cockpit 风格改造，但界面仍存在两个明显问题：

1. 页面上仍保留大量英文文案，且部分历史中文存在编码异常，导致界面语言体验不统一。
2. 全站仅支持暗色主题，缺少明亮主题切换能力，无法覆盖不同场景下的使用偏好。

本次设计目标是在不引入过重框架重构的前提下，为现有 Vue 3 + Pinia 前端补齐：

1. 全站中文/英文切换。
2. 全站暗黑/明亮主题切换。
3. 用户偏好本地持久化。
4. 壳层入口一致、切换即时生效、测试可覆盖。

## 用户目标

### 主要目标

1. 用户进入系统后，可以直接在顶部栏切换中文/英文。
2. 用户进入系统后，可以直接在顶部栏切换暗色/明亮主题。
3. 用户切换后无需刷新页面，当前页和其它页面的文案/样式应立即更新。
4. 用户下次再次访问时，应保持上次选择的语言和主题。

### 非目标

1. 本轮不引入 `vue-i18n` 或其它重型国际化框架。
2. 本轮不做第三种以上主题。
3. 本轮不处理后端返回消息的多语言化，后端原始错误信息仍按现有内容显示。
4. 本轮不建设完整设计系统平台，只建设满足当前项目的轻量偏好与翻译层。

## 方案对比

### 方案 A：轻量自研偏好层 + 轻量字典翻译层

通过 Pinia 新增 `preferences` store 管理 `locale` 和 `theme`，通过一个集中式字典模块提供 `t(key)`，并通过 `data-theme` 驱动 CSS 变量切换。

优点：

1. 与当前代码结构最贴合。
2. 改动可控，能在现有页面上逐步替换硬编码文案。
3. 不会把这轮需求拖成国际化框架迁移。

缺点：

1. 需要手工把现有文案迁移到字典。
2. 若未来支持更多语言，仍需要进一步演进。

### 方案 B：引入 `vue-i18n` + 自研主题层

优点：

1. 长期多语言扩展更标准。
2. 组件内翻译模式更成熟。

缺点：

1. 当前项目没有 i18n 基础，接入改动面更大。
2. 这轮核心目标是快速统一语言体验，不需要立刻承担框架迁移成本。

### 方案 C：完整配置中心化（i18n + 主题 token + 页面级文案配置）

优点：

1. 长期最整齐。

缺点：

1. 当前明显过重。
2. 会把这轮需求从“体验补齐”扩展成“前端基础设施重构”。

## 结论

采用 **方案 A**。

原因：

1. 当前项目已经存在大量分散的组件级文案，轻量字典方式最容易逐步替换。
2. 主题 token 已集中在 `frontend/src/styles/theme.css`，天然适合扩展为双主题。
3. 可以优先交付用户最关心的语言/主题切换体验，并保持后续可演进空间。

## 体验设计

## 壳层入口

语言切换和主题切换都放在顶部栏右侧，与“新建申请”“退出登录”同一操作区。

### 语言按钮

1. 默认显示当前语言：中文模式显示 `中文`，英文模式显示 `EN`。
2. 点击后在 `zh-CN` 和 `en-US` 间切换。
3. 切换后当前页标题、导航、按钮、表格列头、筛选标签等立即更新。

### 主题按钮

1. 默认显示当前主题：暗色模式显示 `暗色`，明亮模式显示 `明亮`。
2. 点击后在 `dark` 和 `light` 间切换。
3. 切换后页面背景、文字、面板、边框、按钮等立即更新。

### 移动端行为

1. 不额外创建隐藏式主题/语言面板。
2. 仍保留在顶部栏可见区域。
3. 通过缩短按钮文案、压缩内边距和保持单行布局避免挤爆。

## 默认与持久化规则

首次进入系统时采用以下优先级：

1. 若 `localStorage` 中已有保存值，则优先使用保存值。
2. 若无保存值，则默认语言为 `zh-CN`，默认主题为 `dark`。
3. 浏览器语言和系统主题仅作为兜底信号保留在初始化逻辑中，以便未来演进，但不覆盖显式保存值。

一旦用户手动切换：

1. 立刻更新当前运行状态。
2. 立刻写入 `localStorage`。
3. 后续页面跳转和再次打开页面都沿用该偏好。

## 信息架构与代码边界

## 新增模块

### `frontend/src/stores/preferences.ts`

职责：

1. 保存当前 `locale` 和 `theme`。
2. 负责初始化、切换、持久化。
3. 对外暴露：
   - `locale`
   - `theme`
   - `initializePreferences()`
   - `setLocale()`
   - `toggleLocale()`
   - `setTheme()`
   - `toggleTheme()`

### `frontend/src/i18n/messages.ts`

职责：

1. 维护 `zh-CN` 与 `en-US` 两套文案字典。
2. 按模块分组 key，例如：
   - `shell.topbar.logout`
   - `shell.topbar.searchPlaceholder`
   - `users.list.create`
   - `users.drawer.titleCreate`
   - `deviceAccounts.list.empty`
3. 提供一致的 key 命名方式，避免继续散落硬编码。

### `frontend/src/i18n/index.ts`

职责：

1. 暴露轻量 `t(key)` 接口。
2. 暴露 `useI18nText()` 之类的组合式封装，供组件内直接读取响应式翻译。
3. 对缺失 key 使用回退逻辑，避免运行期直接崩溃。

### `frontend/src/composables/useTheme.ts`（可选）

若实现过程中需要让主题应用逻辑更清晰，可抽出一个组合函数：

1. 负责把当前主题同步到 `document.documentElement.dataset.theme`。
2. 负责初始化时的 DOM 应用。

若逻辑足够轻，也可直接在 `preferences` store 中处理，无需额外文件。

## 现有模块的改造原则

### `navigation.ts`

当前 `label` 和 `description` 直接存最终显示文本。改造后不再存最终文案，改为存可翻译 key，例如：

1. `labelKey`
2. `descriptionKey`

Sidebar 与 Topbar 使用 key 运行时翻译，而不是读取静态字符串。

### `router meta`

当前 `meta.title`、`meta.subtitle`、`meta.section` 中直接存文案。改造后改为：

1. `titleKey`
2. `subtitleKey`
3. `sectionKey`

Topbar 上下文标题运行时根据 key 翻译，确保切换语言时无需重建路由即可刷新显示。

### 页面与抽屉组件

所有可见文案迁移到字典，包括：

1. Hero 标题与副标题。
2. 按钮。
3. 表格列头。
4. 空状态。
5. 筛选标签。
6. 本地错误提示与提交中状态。
7. `window.confirm` 的本地确认文案。
8. 枚举映射，例如 `Enabled/Disabled`、`Active/Leave/Resigned`、`Bound/Unbound`。

## 主题系统设计

## Token 结构

现有 `theme.css` 已使用 `--cockpit-*` 变量。保持变量名不变，扩展成双主题结构：

1. `:root[data-theme="dark"]`
2. `:root[data-theme="light"]`

这样现有绝大多数页面样式无需改写选择器，只需继续消费统一 token。

## 暗色主题

保留当前 Cockpit 主视觉方向：

1. 深色背景。
2. 冷青 + 蓝色高光。
3. 半透明面板。
4. 偏高对比文字。

## 明亮主题

在不丢失 Cockpit 品牌气质的前提下改成更明快的工作台风格：

1. 背景改为浅灰白渐变与低饱和冷色氛围。
2. 面板底色改为接近白色。
3. 边框改为低对比灰蓝线条。
4. 正文文字改为深灰蓝。
5. 保留青色/蓝色作为主行动与高亮色。

### 不变项

1. 组件层级关系。
2. 圆角与阴影等级。
3. 页面结构与信息布局。

### 需要额外注意的地方

1. 当前部分组件直接写了 `rgba(255,255,255,...)` 或深色叠层色，这些在明亮主题下会失真。
2. 本轮会优先把壳层和核心表单/列表中影响明显的背景、边框、文字、输入框样式替换成 token 驱动。
3. Element Plus 基础控件需要补充一层主题变量覆盖，避免出现页面亮色、控件暗色的割裂感。

## 实施范围

本轮按“用户实际看到的前台文案”全覆盖：

1. 登录页。
2. 顶栏、侧边栏、筛选面板、PageHero。
3. Dashboard。
4. 用户管理页与抽屉。
5. 部门管理页与抽屉。
6. 设备账号页与抽屉。
7. 申请列表页。
8. 新建申请页。

10. 审计日志页。

## 错误处理与回退

### 翻译缺失

若某个 key 在当前语言缺失：

1. 优先回退到中文。
2. 若中文也不存在，则回退到 key 本身。
3. 不允许因为翻译缺失导致页面渲染失败。

### 本地存储异常

若 `localStorage` 不可用或读取失败：

1. 使用默认值 `zh-CN + dark`。
2. 不阻塞应用启动。

### DOM 主题应用失败

若无法设置 `data-theme`：

1. 不阻塞页面渲染。
2. 保底仍以默认暗色样式渲染。

## 影响与风险

### 主要风险

1. 文案散落范围较大，容易漏改。
2. 明亮主题下，现有大量白色透明背景写法可能出现层级不清。
3. 旧测试对英文文案的依赖会在切换到默认中文后集中失败。

### 控制手段

1. 先从壳层和关键页面集中抽字典，再批量替换。
2. 主题变量优先复用现有 token 名称，减少样式面扩散。
3. 测试优先覆盖偏好入口和关键页面，保证主链路不回退。

## 实施顺序建议

1. 建立 `preferences` store 与轻量翻译模块。
2. 接入 `theme.css` 双主题 token 与根节点 `data-theme`。
3. 在 Topbar 加入语言/主题切换按钮。
4. 改造 `navigation.ts` 与 `router meta` 为翻译 key。
5. 分批替换页面与抽屉组件中的硬编码文案。
6. 修正单测并补充偏好与切换测试。
7. 跑完整前端验证。

## 结论

本设计采用轻量自研方案，在不引入重型国际化框架的前提下，为现有 AMS2.0 前端补齐：

1. 中英双语切换。
2. 暗黑/明亮主题切换。
3. 用户偏好持久化。
4. 可验证、可扩展的文案与主题边界。

该方案满足当前“尽快统一体验并可上线”的目标，同时为后续继续扩展更多语言或更多主题预留演进空间。
